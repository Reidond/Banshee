# ghostty-vt-sys

Raw FFI bindings to the vendored `libghostty-vt` static library.

This crate is the **FFI quarantine boundary** (SPEC §6.1(1)): it is the *only*
crate permitted to reference the C header or the `vendor/ghostty-vt/include`
path. `term-core` and everything above it consume the safe Rust surface, never
the C ABI.

## What links here

- `build.rs` points the linker at
  `vendor/ghostty-vt/lib/<arch>/ghostty-vt-static.lib`
  (`x64` for `x86_64`, `arm64` for `aarch64`) and links it statically.
- No Zig, no bindgen, no `libclang` runs during a normal build. The vendored
  static lib and the checked-in `src/bindings.rs` make contributor builds
  pure-Rust (SPEC §5 / UC-01).

The static lib is built by Zig, which bundles `compiler_rt`; the MSVC toolchain
supplies the UCRT + `kernel32` by default, so no extra `cargo:rustc-link-lib`
system libs were needed in M0. If undefined NT/CRT symbols ever surface, add
them in `build.rs` (there is a commented placeholder).

### ARM64

`vendor/ghostty-vt/lib/arm64/` is produced by CI, not by contributor machines.
Until it exists, `cargo build`/`cargo test` for `aarch64` fail at **link** time
with a message naming `cargo xtask vendor-vt`. `cargo check
--target aarch64-pc-windows-msvc` still passes (no link step) — this is the
M0 ARM64 gate.

## Regenerating the bindings

`src/bindings.rs` is **generated once and checked in**. Regenerate only when the
pinned ghostty commit changes (see `vendor/ghostty-vt/UPSTREAM`). Requires the
`bindgen` CLI and `libclang` (LLVM); neither is needed for a normal build.

```bash
# Prerequisites (once):
#   winget install LLVM.LLVM         # provides libclang.dll
#   cargo install bindgen-cli        # provides the `bindgen` binary
# Then, from the workspace root, with LIBCLANG_PATH pointing at LLVM's bin dir:
LIBCLANG_PATH="C:/Program Files/LLVM/bin" \
bindgen vendor/ghostty-vt/include/ghostty/vt.h \
  -o crates/ghostty-vt-sys/src/bindings.rs \
  --allowlist-function 'ghostty_.*' \
  --allowlist-type 'Ghostty.*' \
  --allowlist-var 'GHOSTTY_.*' \
  --default-enum-style rust_non_exhaustive \
  --no-layout-tests \
  --raw-line '// @generated by bindgen 0.72.1 over vendor/ghostty-vt/include/ghostty/vt.h (ghostty commit d560c645). DO NOT EDIT BY HAND.' \
  --raw-line '// Regenerate: see crates/ghostty-vt-sys/README.md' \
  -- \
  -I vendor/ghostty-vt/include \
  -DGHOSTTY_STATIC \
  --target=x86_64-pc-windows-msvc
```

Notes:
- `-DGHOSTTY_STATIC` makes the `GHOSTTY_API` macro a no-op (no `__declspec`
  import/export), which is correct for static linking.
- The `static inline` mode helpers in `modes.h` (`ghostty_mode_new`,
  `ghostty_mode_value`, `ghostty_mode_ansi`) have **no exported symbol** and are
  therefore not bound. They are reimplemented in Rust in `src/lib.rs` from the
  header's documented bit layout.
- Wiring this into `cargo xtask` (regeneration + re-pin) is a future task; a doc
  note suffices for M0.
