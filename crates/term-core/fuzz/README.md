# term-core fuzz seam (UC-02 E1)

`cargo fuzz` target `feed` drives `Terminal::feed`/`resize`/`snapshot` with
arbitrary bytes (see `fuzz_targets/feed.rs` for the exact deterministic
chunking/interleaving strategy). A crash here — an FFI panic or access
violation escaping `ghostty_terminal_vt_write` / `ghostty_terminal_resize` —
is the UC-02 E1 blocker criterion.

## Required invocation on Windows: `RUSTFLAGS` is mandatory

The vendored `vendor/ghostty-vt/lib/x64/ghostty-vt-static.lib` is a Zig-built
static library with the **static CRT** (`RuntimeLibrary=MT_StaticRelease`,
pulls in `libcpmt.lib`). `libfuzzer-sys`'s bundled C++ driver defaults to the
**dynamic CRT** (`/MD`). Building the fuzz binary without reconciling the two
fails at link time with `LNK2038` (RuntimeLibrary mismatch) and cascading
`LNK2005` (duplicate `msvcprt`/`libcpmt` symbols).

The fix is to build the fuzz crate itself with the static CRT so both sides
agree:

```powershell
$env:RUSTFLAGS = "-C target-feature=+crt-static"
cargo +nightly fuzz build feed
cargo +nightly fuzz run feed -- -max_total_time=90 -rss_limit_mb=4096
```

**This must be an environment variable, not `fuzz/.cargo/config.toml`.**
`cargo fuzz` sets its own `RUSTFLAGS` internally (for SanitizerCoverage/ASan
instrumentation) before invoking `cargo build`; env vars take precedence over
`.cargo/config.toml`'s `build.rustflags`, so a config-file entry here is
silently discarded. This was verified empirically during M0 Task 5 — see
`.specs/m0-seance/gap-log.md` if a cross-reference is added there, and the
Task 5 report for the failed `.cargo/config.toml` attempt.

This does **not** affect `term-core`'s own normal build/test/clippy — those
link fine against the vendored lib as-is (proven by the crate's green test
suite); the conflict is specific to libfuzzer-sys's C++ runtime linkage.

## ASan runtime DLL (local Windows runs only)

If `cargo fuzz run` exits immediately with `STATUS_DLL_NOT_FOUND`
(`0xc0000135`), the AddressSanitizer runtime DLL
(`clang_rt.asan_dynamic-x86_64.dll`) isn't on `PATH`. It ships with the MSVC
Build Tools, not with rustup. Add it once per shell session:

```powershell
$env:PATH = "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Tools\MSVC\<version>\bin\Hostx64\x64;$env:PATH"
```

(Adjust the MSVC version path to match your install; find it with
`Get-ChildItem "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Tools\MSVC"`.)
GitHub-hosted `windows-latest` runners have this on `PATH` already via the
preinstalled Visual Studio, so `.github/workflows/fuzz-nightly.yml` does not
need to add it explicitly (verified by the nightly job's own successful run;
if that ever regresses, add the same `PATH` prepend there).

## Corpus / artifacts

`corpus/feed/` and `artifacts/feed/` are gitignored (`fuzz/.gitignore`,
generated by `cargo fuzz init`). A crash produces a file under
`artifacts/feed/` named after its input hash — if CI ever finds one, that
artifact is the reproduction case and must be preserved (do not delete it
when triaging; see the main Task 5 report for the "crash = Gap Log blocker"
policy).
